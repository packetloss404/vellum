Vellum

v1 · localhost · no auth

“A durable investigation system where the dossier is the primary surface, not chat.”

“Vellum is for the kind of question that doesn’t belong in a chat window — a consequential decision with several unknowns, where the obvious answer may be dangerous if key facts are wrong. Layoff decisions. Housing decisions. Ransomware payment decisions. FMLA questions. Credit-card debt negotiations after a parent dies with no estate. Any high-stakes situation where a chatbot producing confident output against a mis-framed question is worse than no output at all, because the user acts on it.”

The problem

WHY CHAT IS WRONG FOR DECISIONS

“Durable thinking on consequential questions doesn’t fit in a chat window. The answer a user needs often depends on assumptions the question itself smuggles in — legal framing that may not apply, facts not yet established, a decision model the situation doesn’t actually fit. ‘What percentage should I open credit-card-debt negotiations at?’ has no responsible answer until you’ve confirmed the debt is actually owed, the statute of limitations hasn’t expired, and the collector owns the account. The same structural problem shows up in layoff decisions, housing decisions, ransomware payment decisions, FMLA questions, infrastructure migrations — any high-stakes situation with several unknowns where the obvious answer may be dangerous.”

“Chat, structurally, dumps all the state into one scrolling transcript. A dossier that evolves — typed, stateful, structured — is the shape this problem actually has.”

WHAT MAKES IT DIFFERENT

Three load-bearing behaviors

Features are easy. Product thesis is harder. These three are the ones Vellum won’t compromise.

01 · Premise Challenge

Challenge the premise.

Before answering, audit what the question smuggles in.

On a new dossier, the agent’s first move is almost never to answer the stated question. It produces a record_premise_challenge tool call with the question’s hidden assumptions, a safer reframe, and the evidence required before any answer would be responsible. The premise challenge renders at the top of the dossier, above everything else, and gates substantive work alongside plan approval.

Concrete: a user asks “what percentage should I open credit-card-debt negotiations at for my deceased mother’s accounts?” Vellum refuses to quote a number until it has audited FDCPA validation status, statute of limitations in each relevant state, and whether the surviving children are personally liable under probate law. Quoting a number on the original question could restart the SoL clock. The helpful answer is the one that does the most damage. Pushback is not a feature — it’s the whole product.

02 · Structured Writes

Structured data, not prose.

Every meaningful write is a typed tool call.

Vellum’s agent cannot emit prose directly to the dossier. Every meaningful write goes through a closed set of typed tool calls. Free-text thinking evaporates at turn boundary; what persists is structure: sections with state, needs-input questions, decision points, working theory with confidence, sub-investigations with scope and findings.

This is the opposite of chat. Chat records every word. Vellum records only what survives the filter of “is this a structured update to the case file?” The transcript is gone by design. What remains is the dossier as it evolved — not a log of how it got there.

  • flag_needs_input(question, blocks_section_ids)
  • flag_decision_point(title, options, kind)
  • record_premise_challenge(original_question, hidden_assumptions, safer_reframe, …)

  • summarize_session(what_moved, confirmed, ruled_out, blocked_on)

03 · First-Class State

First-class state beyond ‘running’.

Confident, provisional, blocked. Active, paused, delivered.

Every section carries a state pip: confident, provisional, or blocked. Plan items have status: planned, in-progress, completed, abandoned. Dossiers carry status: active, paused, delivered. Needs-input and decision-points are first-class surfaces, not afterthoughts. A working theory carries a confidence level the user can read at a glance.

This matters because the user is returning after hours away. They need to see not just what the agent found but how sure it is — and what is currently blocked on them. State is a UI affordance, but it’s also the agent’s own scaffolding for what ‘progress’ means. Demotions happen: when a key fact becomes unknown, a confident section drops to blocked. Promotions happen when confirmation lands.

UNDER THE HOOD

How it actually works.

No framework magic. A manual agent loop over the Anthropic Messages API, 24 Pydantic-typed tools, a 30-second scheduler, and SQLite-backed durability. Listed here because judging what’s real in a demo is the point.

The turn loop

Every turn starts with a freshly serialized state snapshot — dossier, sections, open needs-input, open decision-points, budget, recent tool calls — injected as a single system-level message. The model reasons, emits tool calls, and the turn closes. Prose without a tool call evaporates. The state snapshot regenerates from SQLite on every turn, which is what makes pause/resume boring: it’s always a cold start against current truth.

Sleep-mode scheduler

A single asyncio coroutine polls every 30 seconds. It picks up dossiers where wake_pending=1 (someone just resolved a decision-point or answered a needs-input) or wake_at <= now (the agent asked to sleep until a wall-clock time). It starts a work-session. That’s the whole thing. On app boot, a lifecycle reconciliation sweep finds orphaned work-sessions — processes that died mid-turn — and marks them interrupted_crash, returning the dossier to a resumable state.

The agent can call schedule_wake(reason, at) when a real-world interval is the blocker: waiting on a certified-mail response, a collector’s 30-day FDCPA window, a business opening in the morning. User actions trigger reactive wake within one tick. The scheduler never terminates the agent mid-thought — crossing a budget cap surfaces a decision-point the user can steer; it does not kill the run.

WAKE TRIGGERS

  • schedule_wake → wake_at
  • resolve_decision_point → wake_pending=1
  • answer_needs_input → wake_pending=1

reconciliation on boot handles crashes

Sub-investigations

When the plan calls for a narrow, scoped question — “What’s the CFPB enforcement history on this collector?”, “Is there a CA filial-responsibility statute that reaches consumer debt?” — the agent calls spawn_sub_investigation. This runs a scoped sub-agent loop with a narrower tool surface (no plan edits, no new premise challenges), executes synchronously against the parent turn, and returns a return_summary plus concrete findings_section_ids that land back in the parent dossier. Depth is capped at one. Sub-runs are bounded and cheap; parent runs remain coherent.

Stuck escalation and soft-signal budgets

The agent escalates when it notices itself looping or missing progress over a turn window. Tier 1: a reasoning-trail note. Tier 2: a decision-point the user resolves. Tier 3: a recommended-default decision-point that auto-picks if the user doesn’t pick within an interval. Progress-forcing prevents section-edit loops from counting as work.

Budget tracking is soft-signal. Per-session and per-day USD caps and warn fractions are DB-backed and editable in /settings. Crossing a warn fraction surfaces a heads-up. Crossing the cap surfaces a decision-point — “Mark what I have as delivered,” “Keep going,” “Redirect” — it never terminates the agent mid-thought. Every turn is real money; the product treats it that way without being precious about it.
