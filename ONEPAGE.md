# Vellum — project summary

## The problem

Durable thinking on consequential questions doesn't fit in a chat window. The answer a user needs often depends on assumptions the question itself smuggles in — legal framing that may not apply, facts not yet established, a decision model the situation doesn't actually fit. A chatbot that produces confident output against a mis-framed question is worse than no output at all, because the user acts on it. "What percentage should I open credit-card-debt negotiations at?" has no responsible answer until you've confirmed the debt is actually owed, the statute of limitations hasn't expired, and the collector owns the account. The same structural problem shows up in layoff decisions, housing decisions, ransomware payment decisions, FMLA questions, infrastructure migrations — any high-stakes situation with several unknowns where the obvious answer may be dangerous if key facts are wrong.

## The product

Vellum is a **durable investigation system where the dossier is the primary surface, not chat.** The unit is a structured, typed case file an agent works on over hours or days, and that the user returns to on their own schedule. Three load-bearing behaviors define it: first, the agent **challenges the premise** before answering — a typed `record_premise_challenge` tool produces an audit of the question's hidden assumptions, renders at the top of the dossier, and gates substantive work alongside plan approval. Second, the dossier is **structured data, not a chat transcript** — every meaningful write goes through a closed set of typed tool calls (`upsert_section`, `flag_needs_input`, `flag_decision_point`, `update_working_theory`, `spawn_sub_investigation`, `summarize_session`, `mark_investigation_delivered`), so prose the agent emits without a tool call evaporates. Third, the dossier carries **first-class state beyond "running"** — needs-input, decision-points, working theory with confidence, linked investigation questions, section states (`confident | provisional | blocked`), and a "while you were away" session summary that shows the user what moved since their last visit. No pings, no notifications. The dossier is a destination, not a stream.

## How it holds together

Python + FastAPI + Pydantic backend with SQLite; React + TypeScript + Tailwind frontend in a warm, document-like aesthetic. The agent uses the **direct Anthropic Messages API** with a manual agent loop — not the Claude Agent SDK, because the SDK's default toolset and permission model don't fit a product with 10+ Pydantic-derived dossier tools and a server-injected per-turn state snapshot. Default model is `claude-opus-4-7` for the main loop (long-horizon agent work, precise instruction-following); `claude-sonnet-4-6` for intake; `claude-haiku-4-5` reserved for future session-summary synthesis. A **sleep-mode scheduler** (30-second polling asyncio coroutine, SQLite-backed) lets the agent work across real time: it can call `schedule_wake` when a real-world interval is the blocker, user actions (answering a needs_input, resolving a decision_point) trigger reactive wake within one tick, and crash-resume picks up orphaned sessions on boot. Budget tracking is **soft-signal** (the agent never terminates mid-thought; crossing a cap surfaces a decision the user can steer). **Sub-investigations** run as scoped sub-agent loops that return findings synchronously to the parent. A **stuck-escalation** ladder (heads-up → decision point → recommended-default decision point) keeps the agent from spinning silently. Progress-forcing prevents section-edit loops from counting as work. Every meaningful change lands in a `change_log` that drives a plan-diff sidebar the user reads on return — the dossier *as it evolved*, not a transcript of how it got there.

# Vellum Project Description

Vellum is a durable investigation system for consequential decisions where the primary surface is a structured dossier, not a chat transcript. It is built for questions that should not be answered immediately because the user's framing may already contain unsafe assumptions. Instead of treating the prompt as the ground truth, Vellum gives an agent time, structure, and tools to challenge the premise, investigate the underlying facts, surface blockers, and deliver a case file the user can return to later.

The core idea is that many important decisions do not fit well in chat. A user might ask, "What percentage should I offer to settle this credit-card debt?" A normal chatbot may generate a negotiation strategy. Vellum's agent first asks whether that is even the right question: is the debt valid, is it past the statute of limitations, does the collector own the account, and would negotiation accidentally restart liability? In that case, the responsible answer may not be a number at all. It may be to request validation, stop contact, or avoid engaging until facts are established.

Vellum turns that kind of work into a dossier. The dossier contains a premise challenge, working theory, investigation plan, sections, sub-investigations, needs-input blocks, decision points, artifacts, and a final debrief. Each section carries state, such as confident, provisional, or blocked, so the user can distinguish established findings from tentative reasoning. The agent cannot simply ramble into the interface. Meaningful output is written through typed tools, which means the product is structured data first and prose second.

The agent also works over time. A user can leave and return later to see what changed. The right rail shows session summaries, new findings, blocked paths, ruled-out assumptions, and cost. This makes the dossier feel like an evolving case file rather than a disappearing conversation. Vellum is intentionally quiet by default: no constant notifications, no stream of partial thoughts, and no expectation that the user must babysit the agent. The user returns when they are ready and sees the investigation state.

The prototype is built with a Python FastAPI backend, Pydantic models, SQLite persistence, and a React/TypeScript/Tailwind frontend. The agent uses the Anthropic Messages API through a manual loop with Pydantic-derived tool schemas. The system includes plan approval, stuck detection, soft budget controls, session tracking, sub-investigations, and safeguards to prevent duplicate active runs. For the hackathon, the repo includes a no-network fixture demo at `/stress`, so reviewers can see a fully worked dossier without needing an API key or local database.

Vellum is currently a single-user localhost prototype, not a production collaboration platform. Authentication, hosting, mobile polish, notifications, rich-text editing, and multi-user workflows are out of scope. The submission focuses on the product thesis: for high-stakes reasoning, the valuable interface is not a smarter chat box. It is a durable, inspectable investigation object that pushes back before it answers.
